AI setup basics

The minimum you need to know about llm.json to get AI-driven villagers running. For the full operator-level reference, see docs/llm-providers.md — this page only covers "get it working."

You don't need this to play at all

promptworld runs perfectly well with no AI configured — villagers just act on simple built-in behavior instead of model-driven decisions ("reflex-only"). If you delete or never set up llm.json, that's exactly what you get: a working, watchable village, just without the AI layer described on the previous page.

Where the configuration lives

promptworld new automatically writes an llm.json file into your world's save directory with sensible defaults already filled in — you don't have to write one from scratch. It's read once, when the daemon starts; if you edit it, restart the world for the change to take effect:

promptworld stop demo
promptworld start demo

The easiest path: a local model

For everyday villager decisions, promptworld expects an OpenAI-compatible model server running on your own machine — Ollama is the common choice, listening at http://localhost:11434/v1 by default with a model such as gemma4:12b-mlx loaded. Install Ollama, pull a model, make sure it's running, and the default llm.json promptworld already wrote will find it — no editing required for the simplest setup.

Optional: the cloud tier

A handful of higher-value moments — nightly memory digestion, the narrated chronicle, and Metatron's own conversation with you — are worth spending on a higher-quality cloud model instead of the local one. To enable that tier, set an ANTHROPIC_API_KEY environment variable before starting the daemon; the key itself is never written into llm.json, only the name of the environment variable that holds it. Skip this and those moments simply don't run — nothing else breaks.

Checking it's actually working

  • promptworld status demo (and the Metatron pane in the full-screen client) shows you, per configured model, whether it's up, how busy it is, and how much you've spent this month.
  • promptworld llm demo planner "..." fires one test call directly and tells you which model actually answered.
  • promptworld calibrate demo benchmarks your setup and tells you which game speeds are safe to run at with your hardware and model choice — worth running once after any setup change.

If something looks broken

If villagers seem to be getting stuck or the log shows a run of malformed responses, some smaller local models are unreliable at the structured "tool calling" promptworld's AI-driven decisions rely on. There's a per-model fallback setting for exactly this case, covered along with everything else past this "get it working" level — multiple models, custom routing, spending limits, and upgrading an older configuration — in the full operator reference: docs/llm-providers.md. This page intentionally stops here; that guide is where the deeper knobs live.
